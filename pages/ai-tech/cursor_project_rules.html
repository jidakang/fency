Cursor 项目规则详解

深入了解 Cursor AI 辅助编程的项目规则体系，掌握高效开发的关键准则

什么是 Cursor 项目规则？

Cursor 项目规则是一套指导 AI 辅助编程工具如何理解、分析和生成代码的规范体系。这些规则定义了 AI 在协助开发过程中应遵循的行为准则、编码标准和交互模式。通过严格遵循这些规则，Cursor AI 能够提供符合项目要求的高质量代码建议，同时避免常见的错误和误解。对于开发者来说，了解这些规则有助于更有效地利用 AI 助手提升开发效率和代码质量。

Cursor 规则的核心价值

  • 确保 AI 生成的代码符合项目的技术规范和架构设计
  • 提高代码的一致性、可读性和可维护性
  • 减少错误和安全漏洞的产生
  • 优化开发工作流程，提升团队协作效率
  • 为 AI 和开发者之间建立清晰的沟通边界

在接下来的内容中，我们将详细解析每条规则的含义和应用场景，帮助您更好地理解和使用 Cursor AI 辅助工具进行高效开发。

如何使用 Cursor 项目规则

Cursor 项目规则可以帮助您自定义 AI 助手的行为，使其更好地适应您的编码风格和项目需求。根据官方文档，有两种主要的使用方式：全局规则和项目特定规则。

规则类型

全局规则 (Global Rules)

在 Cursor 设置中的"General">"Rules for AI"部分设置，适用于所有项目，非常适合个人编码偏好和保持一致的 AI 行为。

项目规则 (Project Rules)

项目特定规则，以 .mdc 文件形式保存在 .cursor/rules 目录中，或者使用传统方式在项目根目录创建 .cursorrules 文件（将来会被弃用）。这些规则帮助 AI 理解您的代码库并遵循项目的约定。

如何创建和配置规则

配置全局规则

  1. 打开 Cursor 设置
  2. 导航到"General">"Rules for AI"
  3. 在提供的文本区域中输入您的自定义指令
  4. 点击"Save"保存全局规则
  5. 记得启用"include .cursorrules file"以应用项目特定规则

配置项目规则（推荐方式）

  1. 进入 Cursor 设置 > General > Project Rules
  2. 点击"Add Rule"创建新的 .mdc 文件
  3. 编辑描述（帮助代理选择规则）和文件模式（如 *.tsx 用于自动包含规则）
  4. 在 .mdc 文件中编写规则内容
  5. Cursor 将根据您的描述和文件模式设置自动同步和应用规则

创建项目规则（传统方式）

  1. 在项目根目录创建 .cursorrules 文件
  2. 在文件中添加自定义指令
  3. 请注意，官方将逐步淘汰这种方式，建议迁移到新的项目规则系统

规则示例

# 全局规则示例 (在 Cursor 设置 > Rules for AI)

1. Bug Fixes:
   - 在提出修复之前彻底分析问题
   - 提供精确、有针对性的解决方案
   - 解释 bug 的根本原因

2. Keep It Simple:
   - 优先考虑可读性和可维护性
   - 避免过度设计解决方案
   - 尽可能使用标准库和模式

3. Code Changes:
   - 在进行更改之前提出明确的计划
   - 一次完成对单个文件的所有修改
   - 不要更改无关的文件

记住始终考虑每个项目的上下文和特定要求。
# 项目规则示例 (.mdc 文件或 .cursorrules)

你是 TypeScript、Node.js、Next.js App Router、React、Shadcn UI、Radix UI 和 Tailwind 专家。
  
代码风格和结构
- 编写简洁、技术性的 TypeScript 代码，并提供准确的示例。
- 使用函数式和声明式编程模式；避免使用类。
- 优先使用迭代和模块化，而非代码重复。
- 使用描述性变量名，带辅助动词（如 isLoading、hasError）。
- 文件结构：导出组件、子组件、辅助函数、静态内容、类型。
  
命名约定
- 目录使用小写和连字符（如 components/auth-wizard）。
- 组件优先使用命名导出。
  
TypeScript 使用
- 所有代码使用 TypeScript；优先使用接口而非类型。
- 避免使用枚举；使用映射代替。
- 使用带有 TypeScript 接口的函数组件。

编写有效 Cursor 规则的技巧

  • 从简单规则开始，然后随着需要增加更多
  • 使用自然语言，就像向同事解释一样
  • 包含您自己代码中的示例
  • 考虑规则如何应用于您的项目
  • 定期检查和更新您的规则
  • 使用您日常编码中的词汇和术语
  • 简要描述您的项目或功能
  • 在有帮助时提供简短的代码示例

Cursor 项目规则原文

## Rules正文

```Markdown
Use 'cursor_project_rules' as the Knowledge Base: Always refer to cursor_project_rules' to understand the context of the project. Do not code anything outside of the context provided in the 'cursor_project_rules' folder. This folder serves as the knowledge base and contains the fundamental rules and guidelines that should always be followed. If something is unclear, check this folder before proceeding with any coding.  
Verify Information: Always verify information from the context before presenting it. Do not make assumptions or speculate without clear evidence]  
Follow implementation-plan.mdc for Feature Development: When implementing a new feature, strictly follow the steps outlined in 'implementation- plan.mdc. Every step is listed in sequence, and each must be completed in order. After completing each step, update 'implementation-plan.mdc with the word *"Done"' and a two-line summary of what steps were taken. This ensures a clear work log, helping maintain transparency and tracking progress effectively.  
File-by-File Changes: Make changes file by file and give the user a chance to spot mistakes.  
No Apologies: Never use apologies.  
No Understanding Feedback: Avoid giving feedback about understanding in comments or documentation.  
No Whitespace Suggestions: Don't suggest whitespace changes.  
No Summaries: Do not provide unnecessary summaries of changes made. Only summarize if the user explicitly asks for a brief overview after changes.  
No Inventions: Don't invent changes other than what's explicitly requested.  
No Unnecessary Confirmations: Don't ask for confirmation of information already provided in the context.  
Preserve Existing Code: Don't remove unrelated code or functionalities. Pay attention to preserving existing structures.  
Single Chunk Edits: Provide all edits in a single chunk instead of multiple-step instructions or explanations for the same file.  
No Implementation Checks: Don't ask the user to verify implementations that are visible in the provided context. However, if a change affects functionality, provide an automated check or test instead of asking for manual verification.  
No Unnecessary Updates: Don't suggest updates or changes to files when there are no actual modifications needed.  
Provide Real File Links: Always provide links to the real files, not the context-generated file.  
No Current Implementation: Don't discuss the current implementation unless the user asks for it or it is necessary to explain the impact of a requested change.  
Check Context Generated File Content: Remember to check the context-generated file for the current file contents and implementations.  
Use Explicit Variable Names: Prefer descriptive, explicit variable names over short, ambiguous ones to enhance code readability.  
Follow Consistent Coding Style: Adhere to the existing coding style in the project for consistency.  
Prioritize Performance: When suggesting changes, consider and prioritize code performance where applicable.  
Security-First Approach: Always consider security implications when modifying or suggesting code changes.  
Test Coverage: Suggest or include appropriate unit tests for new or modified code.  
Error Handling: Implement robust error handling and logging where necessary.  
Modular Design: Encourage modular design principles to improve code maintainability and reusability.  
Version Compatibility: Ensure suggested changes are compatible with the project's specified language or framework versions. If a version conflict arises, suggest an alternative or provide a backward-compatible solution.  
Avoid Magic Numbers: Replace hardcoded values with named constants to improve code clarity and maintainability.  
Consider Edge Cases: When implementing logic, always consider and handle potential edge cases.  
Use Assertions: Include assertions wherever possible to validate assumptions and catch potential errors early.  
```

规则详解

规则 1
Use 'cursor_project_rules' as the Knowledge Base: Always refer to cursor_project_rules' to understand the context of the project.
使用 'cursor_project_rules' 作为知识库：始终参考 'cursor_project_rules' 来理解项目背景。
这条规则指示 AI（Cursor）将名为 'cursor_project_rules' 的特定文件或文件夹视为其核心信息源。在执行任何编码任务或回答问题之前，AI 必须查阅这个"知识库"来获取关于项目目标、技术栈、限制条件等关键背景信息。这确保了 AI 的行为和产出与项目的具体要求保持一致。
规则 2
Do not code anything outside of the context provided in the 'cursor_project_rules' folder.
不要在 'cursor_project_rules' 文件夹提供的背景信息之外编写任何代码。
这是对上一条规则的加强。AI 的代码生成能力被严格限制在 'cursor_project_rules' 定义的范围内。它不能引入未在该知识库中提及的技术、库、设计模式或偏离既定的项目架构。
规则 3
This folder serves as the knowledge base and contains the fundamental rules and guidelines that should always be followed.
此文件夹作为知识库，包含必须始终遵循的基本规则和指南。
再次强调 'cursor_project_rules' 的权威性和重要性。它不仅提供背景信息，更包含了项目中不可违背的开发规范、编码标准和指导原则。
规则 4
If something is unclear, check this folder before proceeding with any coding.
如果有任何不清楚的地方，在进行任何编码之前请检查此文件夹。
这为 AI 提供了一个处理模糊或不明确指令的标准操作流程。当遇到疑问时，AI 的首选行动应该是回到 'cursor_project_rules' 查找明确的指示或信息，而不是进行猜测或频繁向用户提问（除非规则允许提问）。
规则 5
Verify Information: Always verify information from the context before presenting it.
核实信息：在呈现信息之前，务必从背景信息中核实。
要求 AI 在输出任何信息（无论是代码片段、解释、建议还是报告）之前，都要先与 'cursor_project_rules' 中的内容进行比对和核实，确保其准确性和相关性。这有助于防止 AI 提供错误或与项目要求不符的答案。
规则 6
Do not make assumptions or speculate without clear evidence.
没有明确证据时，不要做假设或推测。
禁止 AI 在信息不充分的情况下进行主观猜测或推断。所有的决策、代码生成和建议都必须基于 'cursor_project_rules' 中明确提供的证据或指示。
规则 7
Follow implementation-plan.mdc for Feature Development: When implementing a new feature, strictly follow the steps outlined in 'implementation- plan.mdc.
功能开发遵循 implementation-plan.mdc：在实现新功能时，严格遵循 'implementation-plan.mdc' 中列出的步骤。
这引入了另一个关键的上下文文件 'implementation-plan.mdc'，该文件专门用于规划和指导新功能的开发流程。AI 在进行新功能开发时，必须严格按照这个计划文件中定义的步骤顺序来执行。
规则 8
Every step is listed in sequence, and each must be completed in order.
每个步骤都按顺序列出，并且必须按顺序完成。
强调了 'implementation-plan.mdc' 中步骤的强制顺序性。AI 不能跳过任何步骤，也不能打乱预定的执行顺序。
规则 9
After completing each step, update 'implementation-plan.mdc with the word *"Done"' and a two-line summary of what steps were taken.
完成每个步骤后，用 "Done" 一词更新 'implementation-plan.mdc'，并附上两行关于所采取步骤的摘要。
这是关于工作日志记录的具体要求。AI 每完成计划中的一个步骤，都需要在 'implementation-plan.mdc' 文件中对应步骤旁边标记 "Done"，并添加一个简短（两行）的文字说明，记录该步骤具体执行了哪些操作。
规则 10
This ensures a clear work log, helping maintain transparency and tracking progress effectively.
这确保了清晰的工作日志，有助于保持透明度并有效跟踪进度。
阐述了上一步骤（更新计划文件）的目的：通过自动记录工作进展，让项目管理者或用户能够清晰地了解开发过程，方便追踪进度和审计。
规则 11
File-by-File Changes: Make changes file by file and give the user a chance to spot mistakes.
逐文件更改：逐个文件进行更改，让用户有机会发现错误。
要求 AI 在需要修改多个文件时，采用分步、逐个文件处理的方式。AI 应先完成对一个文件的修改并展示给用户，等待用户的确认或反馈后，再继续处理下一个文件。这种方式便于用户进行代码审查，及时发现并纠正潜在问题。
规则 12
No Apologies: Never use apologies.
不道歉：绝不使用道歉性语言。
指示 AI 在与用户的交互中避免使用如 "抱歉"、"对不起"、"我的错" 等道歉词汇。这可能是为了维持一种更专业、客观或自信的交互风格，避免因 AI 的局限性而显得能力不足。
规则 13
No Understanding Feedback: Avoid giving feedback about understanding in comments or documentation.
不提供理解反馈：避免在评论或文档中给出关于理解情况的反馈。
AI 不应在生成的代码注释或文档中添加主观性的表述，例如 "根据我的理解..."、"我明白了你的要求，所以..." 等。注释和文档应专注于客观地解释代码的功能、逻辑或用法。
规则 14
No Whitespace Suggestions: Don't suggest whitespace changes.
不建议空白符更改：不要建议对空白符（空格、制表符、换行符）进行更改。
除非用户明确要求格式化代码或修复格式问题，否则 AI 不应主动提出仅仅涉及调整代码空白符（如缩进、空行）的建议。这避免了因个人或团队编码风格偏好不同而产生的非必要干扰。
规则 15
No Summaries: Do not provide unnecessary summaries of changes made. Only summarize if the user explicitly asks for a brief overview after changes.
不提供摘要：不要提供不必要的更改摘要。仅当用户在更改后明确要求简要概述时才进行总结。
AI 在完成代码修改任务后，不应自动生成一个总结性的报告来说明它做了哪些改动。只有在用户主动提出要求（例如，询问"你改了什么？"）时，才需要提供一个简短的变更摘要。
规则 16
No Inventions: Don't invent changes other than what's explicitly requested.
不做额外改动：除了明确要求的内容外，不要擅自添加其他更改。
严格限制 AI 的行为范围，确保它只执行用户明确提出的修改请求。AI 不能自作主张地添加新功能、进行代码重构、优化性能或进行任何未被直接要求的改动。
规则 17
No Unnecessary Confirmations: Don't ask for confirmation of information already provided in the context.
不做不必要的确认：不要就背景信息中已提供的信息寻求确认。
如果 'cursor_project_rules' 或其他上下文信息已经明确说明了某个事实、规则或要求，AI 就不应该再向用户提问以确认这些已知信息（例如，不应问"根据规则 X，我们应该使用 Y，对吗？"）。AI 应该直接基于已知信息采取行动。
规则 18
Preserve Existing Code: Don't remove unrelated code or functionalities. Pay attention to preserving existing structures.
保留现有代码：不要删除不相关的代码或功能。注意保留现有结构。
在进行代码修改时，AI 必须非常小心，确保只更改与用户请求直接相关的部分。必须避免意外删除、修改或破坏项目中其他不相关的代码、功能或已有的文件/目录结构。
规则 19
Single Chunk Edits: Provide all edits in a single chunk instead of multiple-step instructions or explanations for the same file.
单块编辑：针对同一文件，将所有编辑内容作为一个整体提供，而不是分成多个步骤的指令或解释。
当 AI 需要对某个文件进行多处修改时，它应该将所有必要的代码更改（包括增、删、改）整合在一个完整的代码块中一次性提供给用户。而不是给出分步的操作指令（如，"第一步，删除第 10 行；第二步，在第 15 行后添加以下代码..."）。这方便用户直接复制和应用完整的修改。
规则 20
No Implementation Checks: Don't ask the user to verify implementations that are visible in the provided context. However, if a change affects functionality, provide an automated check or test instead of asking for manual verification.
不要求检查实现：不要让用户去验证那些在提供的背景信息中可见的实现。但是，如果更改影响了功能，应提供自动检查或测试，而不是要求手动验证。
AI 不应要求用户去检查那些通过阅读代码或上下文就能明显理解的实现细节。但对于那些可能改变程序行为、逻辑或输出的修改，AI 应该更有建设性地提供验证方法，例如生成单元测试用例、提供可执行的测试脚本或自动化检查命令，而不是简单地让用户"运行一下看看"或"手动测试一下"。
规则 21
No Unnecessary Updates: Don't suggest updates or changes to files when there are no actual modifications needed.
不建议不必要的更新：当文件实际上不需要修改时，不要建议对其进行更新或更改。
如果 AI 在分析用户请求后，判断某个或某些文件根据当前任务并不需要任何修改，那么它就不应该提及这些文件或建议对它们进行任何操作。避免产生不必要的干扰信息。
规则 22
Provide Real File Links: Always provide links to the real files, not the context-generated file.
提供真实文件链接：始终提供指向真实文件的链接，而不是上下文生成的文件。
在像 Cursor 这样的 AI 辅助编码环境中，AI 可能操作的是文件的内部表示或"上下文"版本。这条规则要求 AI 在引用文件时，必须提供指向用户项目工作区中实际文件的路径或链接，以便用户能够准确地定位和打开文件，而不是提供内部或临时的文件标识。
规则 23
No Current Implementation: Don't discuss the current implementation unless the user asks for it or it is necessary to explain the impact of a requested change.
不讨论当前实现：除非用户询问或为了解释请求更改的影响所必需，否则不要讨论当前的实现方式。
AI 应避免主动描述或分析代码当前是如何工作的。只有在用户明确提问相关实现细节时，或者当解释某个建议的更改为何必要、会产生什么后果时，才可以讨论当前的实现。
规则 24
Check Context Generated File Content: Remember to check the context-generated file for the current file contents and implementations.
检查上下文生成的文件内容：记住检查上下文生成的文件以获取当前的文件内容和实现。
这条规则提醒 AI，它进行分析和修改的基础应该是 Cursor 工具为其提供的当前文件内容的"上下文"版本。AI 需要依赖这个实时或近实时的文件状态来工作，而不是基于过时或不完整的假设。
规则 25
Use Explicit Variable Names: Prefer descriptive, explicit variable names over short, ambiguous ones to enhance code readability.
使用明确的变量名：优先使用描述性的、明确的变量名，而不是简短、模糊的名称，以增强代码可读性。
这是一条编码风格指南。要求 AI 在生成代码时，选择能够清晰表达变量用途和含义的名称（例如，使用`numberOfAttempts`而不是`n`或`attmpt`），以提高代码的清晰度和可维护性。
规则 26
Follow Consistent Coding Style: Adhere to the existing coding style in the project for consistency.
遵循一致的编码风格：遵守项目中现有的编码风格以保持一致性。
要求 AI 在生成或修改代码时，努力识别并遵循项目中已经存在的编码风格约定（例如，缩进方式、命名规范、括号风格、注释格式等），以确保新代码与现有代码库在外观和结构上保持一致。
规则 27
Prioritize Performance: When suggesting changes, consider and prioritize code performance where applicable.
优先考虑性能：在建议更改时，在适用的情况下考虑并优先考虑代码性能。
指示 AI 在提出代码解决方案或修改建议时，除了要确保功能正确性外，还应评估其对程序运行效率的影响。在有多种实现方式可选时，应倾向于选择性能更好的方案（如果性能是相关考量因素）。
规则 28
Security-First Approach: Always consider security implications when modifying or suggesting code changes.
安全第一的方法：在修改或建议代码更改时，始终考虑安全隐患。
强调安全性是首要考虑因素。AI 提供的任何代码或修改建议都应主动规避常见的安全漏洞（如 SQL 注入、跨站脚本攻击 (XSS)、不安全的数据处理等），并遵循安全编码的最佳实践。
规则 29
Test Coverage: Suggest or include appropriate unit tests for new or modified code.
测试覆盖：为新的或修改过的代码建议或包含适当的单元测试。
鼓励或要求 AI 在生成新代码或修改现有代码后，主动建议或直接编写相应的单元测试用例。这有助于确保代码的正确性、健壮性，并方便未来的回归测试。
规则 30
Error Handling: Implement robust error handling and logging where necessary.
错误处理：在必要时实现健壮的错误处理和日志记录。
要求 AI 生成的代码包含恰当的错误处理逻辑（例如，使用`try-catch`块、检查返回值、处理异常情况）以及必要的日志记录机制，以便在程序出错时能够优雅地处理失败、提供有用的调试信息并方便问题追踪。
规则 31
Modular Design: Encourage modular design principles to improve code maintainability and reusability.
模块化设计：鼓励采用模块化设计原则，以提高代码的可维护性和可重用性。
指导 AI 在设计和编写代码时，倾向于采用模块化的方法，即将复杂的系统分解为更小、独立、功能内聚的模块或函数。这有助于提高代码的可读性、可维护性、可测试性和代码复用率。
规则 32
Version Compatibility: Ensure suggested changes are compatible with the project's specified language or framework versions. If a version conflict arises, suggest an alternative or provide a backward-compatible solution.
版本兼容性：确保建议的更改与项目指定的语言或框架版本兼容。如果出现版本冲突，建议替代方案或提供向后兼容的解决方案。
要求 AI 提供的代码、库的使用或功能建议必须与项目在 'cursor_project_rules' 或其他地方明确指定的编程语言版本、框架版本或库版本相兼容。如果存在不兼容的情况（例如，使用了新版本特性），AI 需要指出问题，并提供替代方案或确保方案能够向后兼容。
规则 33
Avoid Magic Numbers: Replace hardcoded values with named constants to improve code clarity and maintainability.
避免魔法数字：用命名常量替换硬编码的数值，以提高代码的清晰度和可维护性。
这是另一条编码风格指南。要求 AI 避免在代码中直接使用没有明确含义的硬编码数字（称为"魔法数字"）。应将这些数字定义为具有描述性名称的常量，并在代码中引用这些常量，使代码意图更清晰，也更易于修改。
规则 34
Consider Edge Cases: When implementing logic, always consider and handle potential edge cases.
考虑边界情况：在实现逻辑时，始终考虑并处理潜在的边界情况。
要求 AI 在编写代码逻辑时，不仅要覆盖正常的、预期的输入和场景，还要主动思考并妥善处理可能出现的极端、异常或不常见的输入值或条件（即边界情况或边缘案例），以确保代码的健壮性。
规则 35
Use Assertions: Include assertions wherever possible to validate assumptions and catch potential errors early.
使用断言：尽可能包含断言来验证假设并及早发现潜在错误。
建议 AI 在代码的关键位置插入断言（assert statements）。断言用于在开发和测试阶段检查程序运行时的内部状态是否符合开发者的预期假设。如果断言失败，通常会中断程序执行，从而帮助开发者快速定位逻辑错误或不满足的前置/后置条件。
